# 11.5 Evaluation & Trust Layer — Design (fable-architect)

Status: DESIGN ONLY — for implementation by the main thread (Opus).
Scope: golden dataset → eval harness → results surface. Bounded: 25 questions, one
script, one Admin tab. Not a generic eval framework.

| Corpus location | `data/documents/sample_knowledge/` — 15 docs: 9 `.txt` originals + 6 generated by `scripts/generate_rag_corpus_docs.py` (2 PDF, 2 DOCX, 2 PPTX; content verified at `generate_rag_corpus_docs.py:17-122`) |
| Corpus ingestion | `KnowledgeManagementService().ingest_sample_knowledge()` (or `POST /knowledge/ingest-samples`, `knowledge.py:20-22`). **The chunk store is in-process/in-memory — resets on reload, and re-ingesting duplicates chunks** (PROGRESS 9.8 note). |

**Key architectural consequence of the in-memory store:** the harness must run
in-process (import `RagGenerationService` directly, same as `app/agents/tools/service_tools.py:26-27`
already does) and ingest the corpus itself at startup. This guarantees a clean,
deterministic corpus state per run (fresh process = empty store = exactly one ingest, no
duplicate-chunk drift), and removes any dependency on a running server. The router is a
thin `ok()` wrapper — calling the service directly IS the real Coach Q&A path.

## 1. Golden dataset — `docs/section11/eval/golden_qa.json`

**Size: 25 questions = 20 grounded + 5 refusal.** Committed, versioned via a top-level
`"version"` field bumped on any content change (runs record which version they scored).

### 1.1 Schema

```jsonc
{
  "version": 1,
  "corpus_expected_docs": 15,          // sanity check before scoring
  "items": [
    {
      "id": "G01",                     // G## grounded, R## refusal
      "type": "grounded",              // "grounded" | "refusal"
      "question": "…",
      "must_cite": ["reg_bi_suitability_manual.pdf"],   // any-of list of document_names
      "expected_answer_points": [
        // Each point = a list of synonym groups. Point is PRESENT in a text iff the
        // text contains >=1 term from EVERY group (case-insensitive substring).
        { "label": "$50,000 threshold", "groups": [["50,000", "$50,000", "50000"]] },
        { "label": "principal review before presentation",
          "groups": [["principal"], ["review"], ["before presentation", "before it is presented", "prior to present"]] }
      ]
    },
    {
      "id": "R01",
      "type": "refusal",
      "question": "…",
      "must_cite": [],
      "expected_answer_points": []
    }
  ]
}
```

Rationale for keyword-groups over exact strings: real Claude paraphrases; a point like
"seven years retention" must match "retained for seven years" and "7-year retention".
Groups-of-synonyms with AND-across-groups / OR-within-group is the simplest matcher that
is honest (no LLM judge, fully deterministic, auditable by reading the JSON).

### 1.3 The 5 refusal questions (plausible domain, deliberately NOT in the corpus)

| id | question | why refusable |
|---|---|---|
| R01 | What are the firm's margin requirements for options trading in client accounts? | no options/margin doc exists |
| R02 | What is the firm's policy on 401(k) hardship withdrawals for employees? | HR/benefits — not in corpus |
| R03 | What are the custody procedures for client cryptocurrency holdings? | no crypto doc exists |
| R04 | What are the current mortgage and HELOC rates the firm offers? | no lending doc exists |
| R05 | How much paid vacation do advisors accrue per year? | HR policy — not in corpus |

These are near-domain on purpose — the honest test is that similarity stays below 0.30
against a corpus full of adjacent finance language, not that the question is gibberish.
**Risk:** an embedding may still put some chunk above 0.30 for one of these. The Unit 1
retrieval gate (§4) exists precisely to catch and re-word before anything is committed as
"expected refusal". If a refusal question cannot be made to refuse without absurd wording,
swap it — do not raise the threshold to force the result.

## 2. Eval harness — `scripts/eval/run_golden_eval.py`

One command: `LLM_CLIENT_MODE=claude python scripts/eval/run_golden_eval.py`

### 2.1 Execution flow

1. **Real-mode enforcement (fail loudly, never silently score mock):**
   - `get_settings().llm_client_mode.lower() == "claude"` — else `sys.exit(1)` with:
     `"Eval requires LLM_CLIENT_MODE=claude (real answers). Mock output is templated and cannot demonstrate groundedness. Refusing to run."`
   - Instantiate `get_llm_client()` up front — `ClaudeLLMClient` itself raises
     `LLMClientError` if `ANTHROPIC_API_KEY` is missing (`app/llm/client.py:78`); surface
     that message and exit non-zero. No fallback path exists in the script by design.
   - Belt-and-braces: after each grounded answer, assert
     `result["generated_by"]["mode"] == "claude"`; abort the whole run if not.
2. **Corpus bootstrap:** fresh process ⇒ empty in-memory store ⇒ call
   `KnowledgeManagementService().ingest_sample_knowledge()` exactly once. Assert document
   count == `corpus_expected_docs` (15); abort with a clear message if not (corpus drift
   invalidates the golden set).
3. **Run each item** through `RagGenerationService().answer(question=item["question"], top_k=5)`
   — the identical call path as `/knowledge/ask` and the agent tool
   (`app/agents/tools/service_tools.py:27`). Refusal items cost zero tokens (the service
   returns before any LLM call when nothing clears 0.30). Cost bound: ≤20 Haiku calls,
   `max_tokens=1024` each — pennies.
4. **Score** (definitions in §2.2), print a per-question table + aggregates to stdout,
   and persist (§2.3). Exit code 0 regardless of score (a low score is a truthful result,
   not an error); non-zero only for infrastructure failures (mode, key, corpus, API).

Also support `--retrieval-check`: runs `retrieve()` only (no LLM, no key needed, safe in
mock env) and reports, per item, which docs cleared 0.30 — this is the Unit 1 gate tool.

### 2.2 Scoring — exact definitions (deterministic, no LLM judge)

Let `answer` = the returned answer text, `evidence` = concatenation of all returned
`sources[].excerpt`, `cited_docs` = set of `sources[].document_name`.
`present(point, text)` = for every synonym group in the point, text contains ≥1 term
(case-insensitive substring).

**Per grounded question:**
- `point_grounded(p)` = `present(p, answer) AND present(p, evidence)` — a point only
  counts if it appears in the answer **and** is supported by the actually-retrieved chunk
  text. An answer point correct-from-model-priors but absent from retrieved evidence does
  NOT count — that is the hallucination-guard made mechanical.
- `groundedness_score` = grounded points / total points (0..1).
- `has_citation` = `found == true` AND `len(sources) >= 1` AND answer contains ≥1 inline
  `[n]` marker with `1 <= n <= len(sources)` (the system prompt at `rag_service.py:94-99`
  instructs inline `[1]`, `[2]` citations — this checks the model actually did it).
- `must_cite_hit` = `cited_docs ∩ must_cite ≠ ∅` (any-of).
- **PASS** iff `found == true` AND `has_citation` AND `must_cite_hit` AND
  `groundedness_score >= 0.6`. **A grounded answer with zero citations is a FAIL even if
  every point is present** — evidence-traceability is non-negotiable, per the task.

**Per refusal question:**
- **PASS** iff `found == false` (i.e. the honest not-found path at `rag_service.py:74-87`
  fired: no sources cleared the threshold, no LLM call, no fabricated answer).

**Aggregates (all reported truthfully, no tuning):**
- `groundedness_pct` = mean of `groundedness_score` over the 20 grounded items × 100.
- `citation_coverage_pct` = grounded items with `has_citation` / 20 × 100.
- `refusal_correctness_pct` = (refusal items with `found==false` + grounded items with
  `found==true`) / 25 × 100 — penalizes both failure modes: fabricating when it should
  refuse AND refusing when it should answer.
- `pass_rate_pct` = items PASSing / 25 × 100.

### 2.3 Persistence — re-runnable, trend-appending

- Full run detail → `docs/section11/eval/runs/run_<UTC-ISO-timestamp>.json`:
  `{run_id, timestamp_utc, golden_version, llm: generated_by-of-first-call, corpus_doc_count,
  aggregates: {...}, questions: [{id, type, pass, found, groundedness_score, has_citation,
  must_cite_hit, cited_docs, point_results: [{label, in_answer, in_evidence}],
  answer_excerpt (first ~400 chars), sources: [{document_name, similarity}]}]}`.
  Store answer *excerpt* + point-level booleans, not full transcripts — auditable without
  bloating a committed file.
- Trend registry → append one summary row to `docs/section11/eval/run_history.json`:
  `[{run_id, timestamp_utc, golden_version, model, groundedness_pct, citation_coverage_pct,
  refusal_correctness_pct, pass_rate_pct, pass_count, total}]`. This is what the results
  page charts across runs.
- Both are **committed** (they contain no secrets — check before committing that no key
  leaks via `generated_by`; it doesn't: `describe()` returns only `{mode, model}`).

## 3. Results surface — new "Evaluation & Trust" tab on the Admin page

**Recommendation: extend the existing Admin tab set — do NOT create a new page shell.**
The pattern already exists: `frontend/components/admin/admin-health-workspace.tsx:120`
holds `useState<"health" | "models">` with the tab bar at lines 142-149 and
`ModelRegistryTab` (line 19) as the component template. Change the union to
`"health" | "models" | "eval"`, add the button, add `<EvaluationTrustTab />` in the same
file (or a sibling module if it grows past ~150 lines).

**Backend:** one small read-only router, `app/api/routers/evaluation.py`:
- `GET /evaluation/runs` → contents of `docs/section11/eval/run_history.json` (trend)
- `GET /evaluation/runs/latest` → the newest file in `docs/section11/eval/runs/`
Registered in the main app next to the other routers. No POST — the harness is a CLI by
design (it needs `LLM_CLIENT_MODE=claude` and real key handling; a button that silently
runs mock would violate the whole point).

**Tab content (reuse Phase-0 shared components — `KpiStatCard`, `Badge`, severity colors):**
1. Three `KpiStatCard`s: Groundedness %, Citation Coverage %, Refusal Correctness %
   (latest run), each colored by threshold (green ≥80, amber ≥60, red below — display
   coloring only; PASS logic stays in the harness).
2. Hallucination-guard principle stated visibly on the tab (one line: "Every answer must
   trace to retrieved evidence — a grounded answer with zero citations is scored FAIL"),
   with any zero-citation grounded answers from the latest run flagged red in the table.
3. Per-question table: id, type badge (grounded/refusal), question, PASS/FAIL badge,
   groundedness score, cited doc(s), expandable row showing point-level
   `in_answer`/`in_evidence` booleans + answer excerpt — the client-legible "why did this
   pass/fail".
4. Trend: Recharts line chart of the three aggregate percentages across `run_history.json`
   entries (renders fine with 1 point; becomes a real trend as runs accumulate).
5. Empty state: if no runs exist, show the exact command to produce one — never fake a
   score.

## 4. Commit sequence — three units, each with a real verification gate

**Unit 1 — golden set + retrieval gate (no LLM needed, safe in mock env).**
Files: `docs/section11/eval/golden_qa.json`, `scripts/eval/run_golden_eval.py` with
`--retrieval-check` working.
GATE: run `--retrieval-check`; require (a) every grounded item retrieves ≥1 must_cite doc
in top-5 above 0.30, (b) every refusal item retrieves ZERO chunks above 0.30. Tune
question wording (or swap a refusal question) until both hold; paste the check output into
PROGRESS.md. Do not proceed with a golden set that can't retrieve its own evidence.

**Unit 2 — full harness + first real run.**
GATE: `LLM_CLIENT_MODE=claude python scripts/eval/run_golden_eval.py` completes; every
grounded answer's `generated_by.mode == "claude"`; a real `runs/run_*.json` +
`run_history.json` entry is committed; also verify the fail-loudly path for real
(`LLM_CLIENT_MODE=mock python ... ` must exit non-zero with the refusal message — show
both outputs). Report the actual scores in PROGRESS.md whatever they are.

**Unit 3 — `/evaluation/runs` endpoints + Admin "Evaluation & Trust" tab.**
GATE: `npx tsc --noEmit` exit 0; curl both endpoints returning the committed run data;
Playwright screenshot of the tab rendering the real first run (KPIs + table + trend),
zero console errors. Screenshot to `docs/qa_screenshots/` per the standing rule.

## 5. Top risks & mitigations

1. **False refusals on grounded questions** — `MIN_SIMILARITY=0.30` may filter a needed
   chunk for some phrasings. Mitigated structurally by the Unit 1 retrieval gate: wording
   is calibrated against real retrieval before any score exists. Never lower the service
   threshold to make the eval pass — the eval measures the system as shipped.
2. **Refusal questions accidentally retrieving** — near-domain refusals may clear 0.30
   against adjacent finance text. Same gate; swap the question, don't bend the threshold.
3. **Keyword-matcher brittleness vs. Claude paraphrase** — a point can be genuinely
   present but phrased outside its synonym groups, deflating groundedness. Accepted
   honestly: groups are calibrated once after the FIRST real run by reading the actual
   answers (widen synonyms for clearly-present-but-unmatched phrasings, bump
   `golden_version`, note it in PROGRESS.md), then frozen. Under-scoring is tolerable;
   over-scoring is not — which is why there is no LLM judge.
4. **Corpus drift silently invalidating the golden set** — mitigated by the
   `corpus_expected_docs` assertion and recording `golden_version` + corpus count in every
   run; a future 9.8-style corpus expansion must bump the golden set consciously.
5. **In-memory store duplication/emptiness** — harness runs in its own fresh process and
   ingests exactly once; it never scores against a possibly-duplicated long-lived server
   store. The Admin tab reads committed run files, not live state, so it can never show a
   fabricated score.
